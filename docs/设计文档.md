# ClawStation 设计文档

## 1. 文档目的
本文档用于统一 `ClawStation + ClawOps` 当前版本的技术设计，覆盖架构分层、核心模型、数据流、存储布局、执行机制和当前实现约束。

本文档以当前仓库代码为准，并作为后续设计评审和实现调整的技术基线。

## 2. 设计目标

### 2.1 核心目标
* 让实例成为统一主入口
* 不向最终用户暴露 `Deployments` 视图或 target 心智
* 让访问平面与运维平面协作但职责清晰
* 让认证和主机密钥在两条链路中共享
* 让失败可诊断、日志可追溯
* 让本地访问地址和浏览器入口地址分离，以支撑多实例并行

### 2.2 当前版本约束
* ClawStation 为桌面端应用
* ClawOps 为独立可执行运维引擎
* 访问和运维均使用系统 OpenSSH
* 第一阶段部署目标是 `Linux + Docker Engine + SSH`
* 当前不负责 Docker 安装和交互式提权

## 3. 总体架构

### 3.1 组件边界

#### ClawStation
职责：

* 管理实例主记录
* 管理认证配置
* 管理 shared `known_hosts`
* 建立和维护 SSH 隧道
* 管理浏览器入口
* 展示实例状态、诊断、最近操作和日志
* 拉起并调用 ClawOps

定位：
* 面向最终用户的产品层
* 负责功能露出、交互流程、状态解释和结果反馈

#### ClawOps
职责：

* 管理 Host / Profile / Target / Operation
* 执行部署、接管校验、探测、升级、回滚和启停
* 维护 revision 和操作记录
* 通过 `stdio JSON-RPC` 向 ClawStation 提供能力

定位：
* 后台执行和技术支撑层
* 不承担最终用户产品心智

### 3.2 组件关系
当前关系为：

1. ClawStation 启动后管理本地实例与配置
2. 当需要部署或运维时，ClawStation 惰性拉起 ClawOps 子进程
3. ClawStation 通过 JSON-RPC 调用 ClawOps
4. ClawOps 执行远端 SSH / Docker 操作并落盘操作记录
5. ClawOps 通过事件将长操作状态回传给 ClawStation

## 5. 核心模型设计

### 5.1 实例模型
主对象为 `InstanceRecord`。

当前关键字段：

* 身份：`id`、`name`、`environment`
* 归类：`sourceType`、`managementMode`
* 关联：`deploymentTargetId`、`credentialId`
* 访问地址：`localAddress`、`browserEntryAddress`
* SSH：`host`、`sshPort`
* 状态：`status`、`statusDetail`、`lastError`
* 接管校验：`adoptValidationState`、`adoptValidationSummary`、`adoptValidationLogPath`

说明：
* 当前实例仍保留 `sshUser / authMethod / privateKeyPath` 作为兼容缓存字段
* 运行时优先通过 `credentialId` 解析认证信息
* 从产品建模上，`InstanceRecord` 是唯一主业务对象

### 5.2 认证模型
认证主对象为 `CredentialProfile`。

当前关键字段：

* `id`
* `name`
* `sshUser`
* `authMethod`
* `privateKeyPath`

秘密存储不在该对象中：

* SSH 密码通过系统 `keyring`
* 托管私钥优先通过系统 `keyring`
* 兼容文件 `config/shared/credential-secrets.json`

### 5.3 运维模型
ClawOps 当前主要维护以下对象：

* `HostProfile`
* `DeploymentProfile`
* `DeploymentTarget`
* `OperationRecord`
* `CurrentRevision`
* `RevisionSnapshot`

当前关系：

* `DeploymentTarget` 可选绑定 `instanceId`
* `OperationRecord` 支持 `instanceId`
* 因此运维记录已经能够回溯到实例

设计结论：
* 这些对象属于 ClawOps 内部技术模型
* ClawStation 可读取和包装，但不应直接暴露为最终用户对象

## 6. 状态与能力设计

### 6.1 实例状态
当前实现的主要实例状态：

* `idle`
* `preparing`
* `connecting`
* `forwarding`
* `probing`
* `ready`
* `degraded`
* `error`

这些状态主要服务于访问链路和界面反馈。

### 6.2 接管校验状态
adopted 实例额外维护：

* `pending`
* `validated`
* `failed`

约束规则：

* `pending` 和 `failed` 状态下，不允许直接连接或打开浏览器
* 需要先执行重新校验

### 6.3 能力矩阵
当前能力按实例类型分流：

* `managed + deployment_managed`
  * 对全新部署实例：支持连接、打开、诊断、日志、部署、探测、重启、回滚
  * 对非全新部署的托管实例：支持连接、打开、诊断、日志、部署、探测，不开放重启、回滚
* `adopted + access_only`
  * 支持连接、打开、诊断、日志、重新校验
  * 当前不暴露正式部署和回滚入口

产品约束：
* 所有能力入口都应从实例发起
* 不应再定义“用户直接操作 target”的正式交互

## 7. 访问链路设计

### 7.1 双地址模型
当前访问设计将“内部隧道地址”和“浏览器入口地址”分离。

#### 内部隧道地址
* 例：`127.0.0.1:39001`
* 由实例分配并持久化
* 作为 SSH 本地端口转发的真实监听地址

#### 浏览器入口地址
* 例：`127.0.0.1:18789`
* 作为最终对用户暴露的地址
* 可被不同实例接管

### 7.2 连接流程
当前连接流程如下：

1. 基于实例生成连接计划
2. 校验认证秘密是否已保存
3. 校验 shared `known_hosts`
4. 启动系统 `ssh` 本地端口转发
5. 监听并探测本地 HTTP 可用性
6. 更新实例状态并按需打开浏览器

### 7.3 认证接入策略
* 密码：通过 `SSH_ASKPASS` 注入
* 外部私钥：使用现有私钥路径
* 托管私钥：运行时生成临时文件并设置严格权限

## 8. 运维链路设计

### 8.1 ClawOps 集成方式
当前采用子进程 + `stdio JSON-RPC`：

* ClawStation 负责定位并启动 `clawops`
* 首次调用时执行握手
* 后续所有操作通过统一 RPC 入口转发

包装原则：
* ClawStation 调用的是后台技术能力
* 返回结果应被包装为“实例动作结果”，而不是“target 动作结果”

### 8.2 协议特征
* 请求/响应使用 JSON-RPC 结构
* 长操作通过通知事件回传：
  * `operation.started`
  * `operation.progress`
  * `operation.log`
  * `operation.finished`
* 前端通过通知更新 target/instance 的运维状态

### 8.3 并发控制
当前通过 `TargetLock` 对同一 `targetId` 做互斥控制。

含义：

* 同一目标不能并发执行多个正式运维动作
* 锁信息会写入 `data/ops/runtime/locks/`

## 10. 日志与诊断设计

### 10.1 ClawStation 诊断
实例诊断当前聚合：

* 状态信息
* 连接计划
* SSH 连通性
* 主机密钥
* 凭据状态
* 本地端口监听
* HTTP 探测

输出形式：

* 诊断摘要
* 多条检查项
* 诊断日志文本
* 诊断日志文件路径

### 10.2 ClawOps 操作日志
每次正式运维动作至少产生：

* `record.json`
* `journal.ndjson`
* `result.json`

当前查询方式：

* 按 `targetId` 聚合
* 按 `instanceId` 聚合

对外呈现原则：
* 最终用户默认只看实例维度的最近操作和日志
* `targetId` 维度仅作为内部实现和调试手段存在

## 11. 错误与安全设计

### 11.1 安全基线
* `known_hosts` 缺失或不可信时默认阻断
* SSH 密码不写入普通配置 JSON
* 托管私钥不写入普通认证元数据 JSON
* OpenSSH 执行统一开启严格主机密钥校验

### 11.2 错误反馈原则
当前设计强调：

* 失败阶段要尽量清晰
* 失败后要给出日志或诊断入口
* 访问链路与运维链路共享基础错误语义

当前实现已较明确区分：

* 主机密钥错误
* SSH 认证或连接错误
* 浏览器入口绑定失败
* 容器不存在 / 未运行
* 端口不可达
* HTTP 探测失败

## 12. 当前关键流程设计

### 12.1 新建托管实例
1. 前端向导收集认证、Host、Profile 和环境参数
2. 通过 `host.create / profile.create / target.create` 建立技术对象
3. 自动同步生成或更新 `managed` 实例
4. 需要时发起 `operation.deploy`

说明：
* `Host / Profile / Target` 是流程中的后台技术步骤
* 不应上升为用户主视角

### 12.2 接管已有实例
1. 前端创建 adopted 实例草稿参数
2. 通过 `operation.adopt` 执行校验
3. 生成正式 `operationId` 和 journal
4. 成功则标记为 `validated`
5. 失败则允许保存为草稿并记录失败摘要

### 12.3 托管运维
1. 从实例详情读取 `deploymentTargetId`
2. 通过 `operation.deploy / probe` 调用 ClawOps；`restart / rollback` 仅对全新部署实例开放
3. ClawOps 落盘操作记录并发出事件
4. 前端刷新操作状态和日志

说明：
* 实例详情是正式入口
* target 维度只作为内部寻址手段

## 13. 当前设计结论

### 13.1 已稳定的技术结论
* 架构采用桌面端 + 独立运维引擎
* 访问和运维统一使用系统 OpenSSH
* `known_hosts` 与凭据解析在两条链路共享
* 实例是主产品对象，运维对象为支撑层
* 操作记录已经可以回溯到实例
* `Deployments` 不应作为面向用户的正式产品结构

### 13.2 仍在收敛中的技术点
* 实例侧认证缓存字段尚未完全移除
* 生命周期状态模型仍偏向当前实现，而非完整产品状态机
* CLI 能力子集少于 `serve` 模式完整能力
