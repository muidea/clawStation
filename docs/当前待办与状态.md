## 2. 当前结论

### 2.1 已确认

1. `edge` 的 SSH、隧道、聊天、历史记录均可正常工作。
2. `1058` 的 SSH、隧道、HTTP 探测都正常，不是传输层问题。
3. `1058` 当前会返回：

```text
AUTH_TOKEN_MISSING: unauthorized: gateway token missing
(set gateway.remote.token to match gateway.auth.token)
```

4. 这说明 `1058` 当前卡在“共享 Gateway token 获取”，还没有进入自动 pairing。
5. 因此不能把“空 token 也能自动 pairing”当成当前桌面端的默认前提。

### 2.2 当前判断

1. 自动 pairing 不是根问题。
2. 共享 token 来源才是根问题。
3. WebUI 手工 approve 不是当前首选方案。
4. 应先用 `clawcli` 在显式 token 模式下验证宿主机部署是否能自动 pairing。

## 5.1 已落地的实例模型调整

`2026-03-25` 已补第一版“部署信息”建模，并接入桌面端与网关控制面：

1. 实例记录新增：
   - `targetPlatform`
   - `deploymentMode`
   - `controlPlaneMode`
   - `containerName`
2. 新建托管实例向导已新增“部署信息”步骤：
   - 可区分 `container` 与 `host_process`
   - 宿主机场景不再强制要求部署模板
3. `gateway.discover` / `gateway.authorizeDevice` 已优先使用实例上的显式控制面信息：
   - `container` 优先走 `docker_exec`
   - `host_process` 优先走 `host_cli`

当前这部分仍未完成的点：

1. `clawops` 的 deploy/probe/restart/rollback 仍主要围绕容器部署设计。
2. 宿主机托管实例目前重点覆盖聊天链路的 Gateway 控制面，不等于完整的一键部署/运维托管。
3. 下一阶段仍要把 `clawcli` 的显式 token 验证结论回落到桌面端 token 自动发现策略。

## 5.2 `host` 模型收口进度

`2026-03-25` 已完成第一轮“一步到位去 host 化”：

1. `clawconsole`：
   - 新建托管实例向导不再调用 `host.list/create/get`

2. `clawops`：
