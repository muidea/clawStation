# 自动认证匹配设计

更新时间：`2026-03-26`

## 1. 目标

本文档用于收口 `ClawStation + ClawOps` 当前已经验证通过的 OpenClaw 自动认证匹配流程，作为后续复用、扩展和回归排障的基线。

当前目标：

1. 在实例添加完成后即可投入使用。
2. 自动完成共享 token 发现、设备配对、Gateway 握手。
3. 区分容器部署和宿主机进程部署两条控制面。
4. 把失败阶段明确落到 `transport / shared_token / device / gateway`。

## 2. 适用范围

当前已验证通过的范围：

1. 平台：`Linux`
2. 部署方式：
   - `container`
   - `host_process`
3. 接入方式：
   - 已有实例接入
   - 新实例部署后验证

当前验证基线：

1. `edge`
   - Linux
   - Docker 容器 `work-station`
   - 自动 token 发现、自动 pairing、握手可通
2. `1058`
   - Linux
   - 宿主机进程部署
   - 已通过 shell/profile/PATH 修复打通自动认证匹配

## 3. 输入模型

自动认证匹配依赖实例上的部署元信息，不再猜测：

1. `targetPlatform`
2. `deploymentMode`
3. `controlPlaneMode`
4. `containerName`
5. `host`
6. `sshPort`
7. `credentialId`
8. `browserEntryAddress`

设计要求：

1. 这些字段必须在实例创建时明确落盘。
2. 后续 `gateway.discover / gateway.authorizeDevice` 只能基于这些字段分流。

## 4. 总体流程

自动认证匹配按以下阶段执行：

1. `SSH / transport`
2. `OpenClaw Web 探测`
3. `shared token 发现`
4. `Gateway 首次握手`
5. `pairing required` 时自动设备授权
6. `Gateway 重试握手`
7. 标记实例验证通过

目标状态：

1. 向导内验证通过后，实例直接可聊天、可拉历史记录。
2. 不允许“实例创建成功但仍需手工再认证一次”。

## 5. 共享 Token 发现

### 5.1 容器部署

容器部署优先通过容器控制面执行：

1. `openclaw dashboard --no-open`
2. `openclaw config get gateway.remote.token`
3. `openclaw config get gateway.auth.token`

特点：

1. 优先使用 `containerName`
2. 默认执行 `docker exec <container> sh -lc '...'`
3. `dashboard --no-open` 返回的 URL 片段 `#token=...` 可以直接提取共享 token

### 5.2 宿主机进程部署

宿主机部署优先通过 `host_cli` 执行：

1. 探测远端登录 shell
2. 按 `bash / zsh / sh` 选择命令包装
3. 加载对应 profile
4. 注入常见 `PATH / nvm / node` 路径
5. 再执行：
   - `openclaw dashboard --no-open`
   - `openclaw config get gateway.remote.token`
   - `openclaw config get gateway.auth.token`

设计原因：

1. 宿主机的非交互 SSH 环境可能缺少 `node`
2. 不能默认假设 `sh -lc` 与用户手工登录环境一致

### 5.3 本地 HTTP fallback

在本地隧道已就绪时，桌面端还会尝试：

1. 访问当前实例的本地 HTTP 入口
2. 从响应内容中提取 `token=...`

设计原因：

1. 避免完全依赖远端 CLI 输出
2. 在部分目标版本中，dashboard URL 比 config path 更稳定

## 6. 自动设备配对

当共享 token 已可用，但首次握手返回 `pairing required` 时：

1. 调用 `gateway.authorizeDevice`
2. 执行：
   - `openclaw devices list --json`
   - `openclaw devices approve ... --json`
   - `openclaw devices rotate ... --json`
3. 保存返回的 `deviceToken`
4. 再次发起握手

设计约束：

1. `pairing required` 不应直接视为验证失败
2. 必须先尝试自动授权和重试握手

## 7. 混合输出解析

`openclaw ... --json` 的真实输出可能并不纯净，前后可能混入：

1. 插件加载日志
2. 环境告警
3. 额外说明文本

当前处理策略：

1. 不再要求整段 `stdout/stderr` 必须纯 JSON
2. 从混合输出中逐段尝试提取并反序列化第一个有效 JSON 对象

这一点对于容器部署尤其重要，因为插件日志常见于：

1. `devices list --json`
2. `devices rotate --json`

## 8. 向导内验证时机

实例向导当前顺序为：

1. `实例与环境`
2. `主机连接信息`
3. `认证配置`
4. `部署信息`
5. `复核与创建`
6. `连接验证`

设计要求：

1. `复核与创建` 只负责创建实例记录
2. `连接验证` 负责把实例推进到可用状态
3. 在验证阶段，实例会先被置为 `pending`
4. 向导内部允许在 `pending` 状态下继续执行：
   - `connect`
   - `perform_gateway_handshake`
5. 用户侧手动 `connect/open/chat` 仍然禁止 `pending/failed`

## 9. 失败分层

失败必须按阶段归类，而不是统一报“握手失败”。

### 9.1 transport

包含：

1. SSH 隧道启动失败
2. 本地 HTTP probe 失败
3. 本地转发地址未就绪

### 9.2 shared_token

包含：

1. `gateway.discover` 未拿到 token
2. dashboard/config 输出不可解析
3. 共享 token 缺失导致 `AUTH_TOKEN_MISSING`

### 9.3 device

包含：

1. `devices list` 失败
2. `devices approve` 失败
3. `devices rotate` 失败
4. 未找到匹配的 pending request

### 9.4 gateway

包含：

1. 自动 pairing 后重试握手仍失败
2. WebSocket / RPC 层被服务端拒绝

## 10. 当前已修复的关键问题

当前链路已修复的关键问题包括：

1. `shared token` 发现顺序不完整
2. 宿主机非交互环境缺少 `node`
3. 聊天轻量隧道把“session 已注册”误判为“transport ready`
4. `browser entry` 端口被占用时错误地中断整个连接
5. 向导内部 `pending` 状态无法继续握手
6. 向导等待 `ready` 时读取了旧的 React state 闭包
7. `devices list --json` 前缀插件日志导致 JSON 解析失败
8. `连接验证` 路径遇到 `pairing required` 时没有自动 `authorizeDevice`

## 12. 后续复用建议

后续若在其他入口复用自动认证匹配能力，应遵循：

1. 不要单独复制“只握手一次”的简化逻辑
2. 必须复用完整阶段：
   - token 发现
   - pairing
   - retry handshake
3. 不要再以输出“整段纯 JSON”为前提
4. 宿主机执行环境必须经过 shell/profile/PATH 适配
5. 任何新入口都应复用同一套阶段日志命名，便于统一排障
