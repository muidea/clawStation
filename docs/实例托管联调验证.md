# 实例托管联调验证

## 1. 目标

验证当前“实例中心”设计下的托管主路径已经成立：

1. 从 `ClawStation` 创建托管实例。
2. 托管实例与后台托管绑定正确建立。
3. 从实例详情发起 `deploy / probe / restart / rollback`。
4. 操作结果、日志、状态反馈回到实例视角，而不是旧的 `Deployments` / `target` 视角。

## 2. 当前代码前提

当前代码已经具备以下条件：

* `ClawStation` 不再面向用户暴露 `Deployments` 视图。
* 实例详情中的托管动作通过 `instanceId` 发起。
* `Tauri` 命令层会把实例解析到内部 `managedBindingId`，再转发给 `ClawOps`。
* 前端公共模型不再暴露 `deploymentTargetId`。
* 后台存储层使用 `managedBindingId`，并兼容读取旧字段 `deploymentTargetId`。

## 3. 环境准备

### 3.1 本地依赖

在仓库根目录执行前，确认：

* `Node.js` / `npm` 可用
* `Rust` / `cargo` 可用
* `cargo tauri` 或本地 `@tauri-apps/cli` 可用

建议先做基础校验：

```bash
cd ./clawconsole
npm run build
npm run test
cd ./src-tauri
cargo check
```

### 3.2 联调目标机

至少准备一台可访问 Linux 主机，满足：

* 可通过 SSH 连接
* 已安装 Docker
* 具备创建目录、启动容器、查看容器日志权限
* 目标用户对部署根目录有写权限

建议记录以下信息：

* 主机地址
* SSH 端口
* SSH 用户
* 认证方式：密码 / 私钥 / 托管私钥
* OpenClaw 镜像地址
* 健康检查端口与路径

### 3.3 凭据与信任

在 `ClawStation` 中先准备好：

* 认证配置
* 目标主机的 `known_hosts` 信任

如果主机密钥未信任，托管向导和后续运维通常会卡在 SSH 前置检查阶段。

## 4. 启动方式

### 4.1 启动 ClawStation

推荐开发联调方式：

```bash
cd ./clawconsole
npm run tauri:dev
```

也可以使用：

```bash
cd ./clawconsole
make tauri-dev
```

说明：

* `ClawStation` 会在首次调用 `clawops` 能力时自动拉起 `ClawOps` 子进程。
* 一般不需要手工先启动 `clawops serve --stdio`。

### 4.2 单独检查 ClawOps

如果要单独确认 `ClawOps` 可执行文件和帮助信息：

```bash
cd ./clawops
cargo run -- --help
```

如果要手工验证协议服务可启动：

```bash
cd ./clawops
cargo run -- serve --stdio --root /tmp/claw-manual-check
```

这条命令会进入 stdio 服务模式，通常只用于手工协议调试。

## 5. 联调步骤

### 5.1 创建认证配置

在 `ClawStation` 中：

1. 打开认证配置管理。
2. 新建一条可用于目标主机的认证配置。
3. 如果是密码或托管私钥，确认秘密已经写入安全存储。

通过标准：

* 认证配置能保存成功。
* 认证检查状态不是缺失。

### 5.2 新建托管实例

在主界面点击“新建实例”，选择托管实例路径，完成向导：

1. 选择认证配置
2. 选择或创建主机
3. 选择或创建部署模板
4. 填写实例环境
5. 选择“立即部署”或“仅保存”

通过标准：

* 实例列表中出现新实例。
* 新实例 `managementMode` 为托管模式。
* 不需要用户理解 `target/profile/host` 也能完成流程。

如果选择“仅保存”，则继续执行 5.3。

### 5.3 从实例详情发起部署

进入实例详情，点击“部署”。

通过标准：

* 操作从实例详情触发，不依赖任何 `Deployments` 页面。
* 界面提示为“实例运维已受理”或等价实例语义提示。
* 实例操作记录区出现新的部署记录。

失败时重点检查：

* 主机密钥是否已信任
* 认证配置是否可用
* Docker 是否安装且可执行
* 镜像地址是否可拉取

### 5.4 执行探测

部署成功后，在实例详情点击“探测”。

通过标准：

* 能生成一条新的实例操作记录
* 状态、当前步骤、错误码、日志可在实例详情查看
* 提示文案不出现 `targetId` 或 `deployment target`

### 5.5 执行重启

在实例详情点击“重启”。

通过标准：

* 操作从实例详情发起
* 实例操作记录增加 `restart`
* 完成后可再次执行探测确认服务恢复

### 5.6 执行回滚

前提：至少存在一个可回滚的上一个 revision。

在实例详情点击“回滚”。

通过标准：

* 回滚确认框为实例语义
* 回滚操作记录写入实例操作区
* 回滚后探测结果与预期版本一致

## 6. 建议采集的验证证据

联调时建议保留：

* 托管实例创建前后截图
* 实例详情中 `deploy / probe / restart / rollback` 操作截图
* 实例操作记录截图
* `operation.logs` 返回的关键日志
* 如失败，记录错误码、错误信息、当前步骤

## 7. 最小验收结论

满足以下条件，可认为“实例托管”在当前阶段已具备最小联调闭环：

* 能从实例创建向导生成托管实例
* 能从实例详情直接发起部署和探测
* 操作记录和日志回到实例详情展示
* UI 不再要求用户理解 `Deployments`、`target`、`profile`
* 前端主路径不再直接依赖内部托管绑定字段

## 8. 当前限制

当前仍有以下限制：

* `ClawOps` 内部执行主键仍然是 `targetId`
* 联调仍然依赖真实远端环境
* 仓库内尚未补齐自动化 e2e 覆盖“创建托管实例 -> deploy -> probe -> restart/rollback”的完整链路

## 9. 建议后续工作

建议按以下顺序继续：

1. 先执行一次真实联调，并把问题清单补到缺陷列表。
2. 为“托管实例创建 + deploy + probe”补最小自动化集成测试。
3. 再评估是否需要继续把 `ClawOps` 内部 `target` 概念抽象成更中性的托管实体。
