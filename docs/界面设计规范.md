# ClawStation 界面设计规范

## 1. 文档目的
本文档基于参考图 `./openclaw_desktop_dashboard.png` 整理当前桌面控制台的目标界面规范，用于统一后续页面设计、组件实现和视觉评审标准。

本文档不是单页还原说明，而是提炼为可复用的整体设计约束。后续新增页面、重构页面和组件抽象应以本文档为准。

## 2. 目标视觉基线

### 2.1 产品气质
目标界面应保持以下气质：

* 桌面端优先的运维后台
* 清晰、克制、专业，不做营销化视觉
* 轻玻璃感和浅色分层，而不是厚重深色控制台
* 以“可扫描的结构化信息”为主，而不是以装饰性插画或大面积空白为主

### 2.2 风格结论
基于参考图，目标风格应固定为：

* 浅色主题为默认主题
* 冷白到浅蓝的背景渐变
* 面板使用高亮白底、细描边、轻阴影
* 信息密度中等偏高
* 图标、状态、操作入口尽量使用统一语义色，不允许每个模块各自定义视觉语言

### 2.3 当前项目收敛方向
当前 `clawconsole` 已存在较强的深色控制台风格。后续界面演进时，应以本规范作为目标视觉基线，逐步收敛以下差异：

* 默认背景从深色氛围调整为浅色后台基底
* 主要容器从重阴影深色面板调整为轻描边浅色卡片
* 信息组织从“模块堆叠”收敛为“导航 + 摘要 + 主表格/主工作区”
* 所有新增页面不得继续扩散新的视觉分支

## 3. 设计原则

### 3.1 结构先于装饰
优先保证布局秩序、分区关系、标题层级、表格扫描效率，再处理视觉细节。

### 3.2 状态先于颜色
颜色必须服务于状态表达。绿色表示正常，黄色表示处理中或风险，红色表示失败或中断，蓝色用于中性动作和主导航，禁止重新解释语义色。

### 3.3 一屏可理解
桌面端核心页面应在首屏完整呈现：

* 当前所在导航
* 核心概览指标
* 主工作区
* 可执行动作入口

用户不应依赖滚动才能理解页面主任务。

### 3.4 高信息密度但不拥挤
允许展示较多实例和状态，但必须通过固定间距、统一表头、标签化状态和留白来保持可读性。

## 4. 信息架构约束

### 4.1 页面骨架
桌面端主界面采用三段式骨架：

* 左侧固定导航
* 右侧主内容区
* 主内容区内部采用“摘要区 + 主工作区”纵向结构

不建议在主内容区继续出现复杂的多层嵌套侧栏。

### 4.2 导航层级
导航最多两层：

* 一级导航：核心产品模块，如 `Dashboard`、`Instances`、`Auth`、`Logs`、`Settings`
* 二级导航：仅在当前一级模块内部出现，且应局部化，不进入全局侧栏

禁止在左侧全局导航中混入操作按钮、说明文案或大块统计卡片。

### 4.3 首页结构
首页应稳定包含以下区域：

* 页标题或模块标题
* 关键指标卡片
* 主列表或主工作区

摘要卡片不超过 3 到 4 个。首页核心区域必须服务于实例管理，不应被次要功能挤占。

## 5. 布局规范

### 5.1 画布与尺寸
桌面端基准宽度按 `1366px` 附近设计，最小可用宽度不低于 `1280px`。低于该宽度时可以进入紧凑模式，但不以移动端布局思路处理。

建议基线：

* 页面外边距：`16px` 到 `24px`
* 侧栏宽度：`220px` 到 `248px`
* 主内容区分组间距：`24px` 到 `32px`
* 卡片内边距：`20px` 到 `28px`
* 表格行高：`52px` 到 `60px`

### 5.2 网格与对齐
所有块级区域遵循统一左对齐线：

* 页面标题
* 摘要卡片
* 表格标题
* 表格本体

禁止不同模块出现明显错位的起始边线。

### 5.3 圆角体系
圆角应统一，不允许每类组件自定义一套半径。

建议值：

* 一级容器：`20px` 到 `24px`
* 二级卡片：`16px` 到 `20px`
* 输入框/按钮：`12px` 到 `16px`
* 状态标签：胶囊圆角，即 `999px`

## 6. 视觉 Token 约束

### 6.1 背景
背景应为冷白到浅蓝的柔和渐变，允许非常轻的环境光晕，但不得抢夺信息层级。

建议基线：

* 页面背景：`#eef5fb` 到 `#dfeef9` 之间的浅色渐变
* 侧栏背景：比页面主背景略深或略白的半透明浅面
* 主卡片背景：接近纯白

### 6.2 边框与阴影
界面依赖边框和轻阴影建立层级，不依赖厚重投影。

建议基线：

* 边框颜色：低对比浅灰蓝
* 阴影：低模糊、低透明度、下沉距离短
* 禁止使用高对比硬边框或大面积重阴影

### 6.3 文字颜色
文字以深灰蓝为主，而不是纯黑。

建议层级：

* 主标题：深色高对比
* 表头和字段标题：中高对比
* 辅助说明：中低对比
* 禁止出现过浅文字承载主要信息

### 6.4 语义色
语义色使用范围必须固定：

* 蓝色：导航选中、主操作图标、中性系统动作
* 绿色：正常运行、成功
* 黄色：警告、部署中、处理中、待关注
* 红色：停止、失败、错误

语义色应优先落在状态标签、图标底色、关键数据提醒上，不应用于大面积背景。

## 7. 字体与排版规范

### 7.1 排版目标
排版要服务于后台扫描效率，避免过多展示性排版。

### 7.2 层级建议
建议层级如下：

* 页面主标题：`32px` 左右，`600` 到 `700`
* 区块标题：`20px` 到 `24px`，`600`
* 卡片标题：`16px` 到 `18px`，`500` 到 `600`
* 主数据数字：`40px` 到 `48px`，`600` 到 `700`
* 正文和表格内容：`14px` 到 `16px`
* 表头和辅助信息：`13px` 到 `14px`

### 7.3 文本约束
所有标题文案应短而稳定，优先使用名词结构，如：

* `Quick Stats`
* `Instance Management`
* `Logs`

避免将整句说明直接作为标题。说明性文案应放在标题下方，且默认不超过两行。

## 8. 核心组件规范

### 8.1 左侧导航
左侧导航应满足以下约束：

* 使用图标加文字
* 当前项有明确选中背景
* 选中状态以浅蓝底和蓝色图标/文字表达
* 非选中项保持低干扰
* 导航项高度和间距统一

禁止出现：

* 同时使用描边高亮、重阴影高亮、左侧色条高亮三种选中方式
* 每个导航项不同颜色
* 导航区信息过载

### 8.2 摘要卡片
摘要卡片用于表达少量关键指标，必须统一结构：

* 标题
* 图标或图标底板
* 主数字

建议约束：

* 卡片数量控制在一行 3 个或 4 个
* 卡片高度一致
* 数字必须是视觉中心
* 不在摘要卡片中堆叠多个次级指标

### 8.3 主数据表格
表格是该类后台的核心组件，必须优先保证可扫描性。

约束如下：

* 表头固定、语义明确
* 列顺序稳定，不频繁变动
* 状态列放在最前
* 主标识列放在状态后
* 快捷操作列放在最右
* 表格外层应作为一个完整容器出现，而不是多块列表拼接

推荐列结构：

* 状态
* 名称
* IP / Endpoint
* 环境
* 健康度或延迟
* 快捷操作

### 8.4 状态标签
状态必须以胶囊标签承载，而不是只靠纯文本或小圆点。

约束如下：

* 标签颜色使用低饱和浅底 + 中等对比文本
* 同类状态颜色全局一致
* 标签长度控制在单词或短词级别
* 同一行中最多出现一个主状态标签

### 8.5 快捷操作
快捷操作应使用统一图标按钮体系。

约束如下：

* 图标尺寸一致
* 点击区域不小于 `32px`
* 默认使用中性蓝系
* 危险操作不放入默认可见的快捷入口
* 更多操作使用省略号菜单收纳

### 8.6 微图表
若在表格中显示延迟趋势或健康趋势，应作为辅助手段存在：

* 尺寸小而稳定
* 单色或低色数
* 不与主状态争抢注意力
* 无需复杂坐标轴和说明

## 9. 交互状态规范

### 9.1 Hover
悬停反馈应轻量：

* 背景轻微加深或提亮
* 边框略微增强
* 必要时配合极小位移

禁止使用过强的缩放、跳动和发光效果。

### 9.2 Active / Selected
选中态只允许通过以下方式组合表达：

* 背景变化
* 字色变化
* 图标色变化

不建议再叠加粗边、强投影和复杂动画。

### 9.3 Disabled
禁用态必须明确，但仍保持可识别。

约束如下：

* 降低对比度
* 保留轮廓
* 禁止仅依赖透明度低到不可见

### 9.4 Loading / Empty / Error
必须预留统一的系统状态界面：

* Loading：使用轻量骨架屏或行级占位
* Empty：提供一句明确说明和一个下一步动作
* Error：提供状态说明、原因概述和可重试入口

禁止使用无文案的空白区域代替状态反馈。

## 10. 内容与数据展示约束

### 10.1 文案风格
文案应简短、直接、可执行：

* 标题使用名词
* 操作按钮使用动词
* 状态使用短标签
* 错误信息优先给出结果和下一步动作

### 10.2 数据格式
数据展示需要统一：

* 延迟统一为 `ms`
* 数量优先用整数
* IP、实例名、环境名称使用统一大小写规则
* 缺失值统一使用 `-`

### 10.3 扫描优先级
用户进入页面后的视觉扫描顺序应固定为：

1. 当前模块
2. 关键指标
3. 实例状态
4. 快捷动作

任何设计调整都不应打乱这个顺序。

## 11. 实现约束

### 11.1 Token 化要求
后续前端实现必须将以下内容抽成统一 token，而不是散落写死：

* 背景色
* 面板色
* 文字层级色
* 语义色
* 边框色
* 阴影
* 圆角
* 间距

### 11.2 组件复用要求
以下组件应优先形成复用实现：

* `SidebarNav`
* `StatCard`
* `SectionHeader`
* `DataTable`
* `StatusPill`
* `IconActionButton`
* `OverflowMenu`

禁止每个页面重新拼一套视觉近似但细节不一致的私有实现。

### 11.3 页面开发约束
后续页面实现必须满足：

* 先确定页面骨架，再填充模块
* 先复用 token 和基础组件，再写局部样式
* 新增样式不得破坏现有全局层级体系
* 若必须偏离本规范，需在设计评审中说明理由

## 12. 明确禁止项
以下做法视为偏离目标设计语言：

* 在同一页面混用深色控制台和浅色后台
* 卡片圆角、阴影、边框风格各不相同
* 表格和卡片使用不同的排版体系
* 大面积高饱和渐变铺满内容区
* 同类状态使用不同颜色
* 图标风格不统一
* 为了“显得有设计感”增加无业务含义的装饰区块

## 13. 评审清单
界面评审时至少检查以下问题：

* 是否仍然符合“左导航 + 摘要 + 主工作区”的骨架
* 是否保持浅色后台基线
* 是否只使用统一语义色表达状态
* 是否建立了清晰的标题、数字、表格层级
* 是否能在首屏理解当前页面主任务
* 是否存在多套圆角、阴影、边框标准
* 是否优先复用了公共组件和 token

## 14. 后续建议
本文档可作为下一步工作的输入：

* 将 `clawconsole/src/styles.css` 收敛为新的浅色 token 体系
* 先重构首页 Dashboard 和实例列表页
* 再抽离可复用的侧栏、卡片、表格、状态标签和图标按钮组件
