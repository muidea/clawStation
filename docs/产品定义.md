# ClawStation 产品定义

## 1. 文档目的
本文档用于重新核对并统一 `ClawStation + ClawOps` 当前版本的产品定义，作为后续开发、评审和文档维护的产品基线。

本文档以当前仓库实现为主，并吸收已有 PRD、技术设计和运维设计中的稳定结论。

## 2. 产品定位

### 2.1 产品是什么
ClawStation 是一个面向 OpenClaw 实例的桌面化安装、接管、访问和基础运维工具。

它由两个组件组成：

* **ClawStation**
  * 桌面端主应用
  * 负责实例管理、认证配置、SSH 隧道、浏览器入口、诊断和日志查看
* **ClawOps**
  * 配套运维执行引擎
  * 负责部署、接管校验、探测、启停、升级、回滚和操作日志落盘

### 2.4 命名与仓库边界
当前项目按“产品名”和“仓库名”分层管理：

* 产品名统一使用 `ClawStation`
* 聚合仓库目录为 `clawStation/`
* 子仓库目录保持 `clawconsole/` 与 `clawops/`
* `clawconsole` 是桌面端实现仓库名，不等同于最终产品展示名
* `clawops` 是后台运维引擎仓库名

结论：
* 对外产品文案、窗口标题、打包名称统一使用 `ClawStation`
* 对内代码路径、submodule 路径、仓库地址继续保留 `clawconsole` / `clawops`

### 2.5 Windows 发布约定
当前 Windows 交付约定已经固定：

* 仅保留 `x86_64-pc-windows-gnu`
* 默认对外提供 `release` 版验证包
* 标准 zip 中必须同时包含 `clawstation.exe`、`clawops.exe` 和 `WebView2Loader.dll`

详细规则见：

* `docs/Windows发布说明.md`

### 2.2 产品解决的问题
当前产品主要解决以下问题：

* OpenClaw 默认仅监听远端 `127.0.0.1`，访问依赖手工 SSH 隧道
* 多实例并行访问时容易出现本地端口冲突和浏览器登录态混淆
* 用户不希望重复执行 SSH、Docker、探测、日志查看等机械化运维步骤
* 新部署实例和已有实例都需要被纳入统一管理，而不是分裂成两套入口

### 2.3 当前版本产品形态
当前版本已经不是概念原型，而是具备真实功能闭环的产品雏形：

* 支持实例管理
* 支持认证配置管理
* 支持 SSH 隧道连接和浏览器访问
* 支持“部署新实例”
* 支持“接管已有实例”
* 支持实例级诊断、日志和部分运维动作

同时当前版本仍处于“实例中心收敛阶段”：

* 实例已经是主用户对象
* 托管实例和接管实例的能力矩阵已经分化，但还未完全最终定型

## 3. 目标用户

### 3.1 运维 / 开发用户
特点：

* 管理测试、预发、生产等多个 OpenClaw 环境
* 熟悉 SSH 和 Docker
* 希望减少重复操作，保留可诊断性和可追踪性

### 3.2 轻度技术用户
特点：

* 需要偶尔部署、接管或访问 OpenClaw
* 不希望直接处理 SSH 细节、主机密钥和 Docker 命令
* 更依赖桌面端提供默认值、校验和失败反馈

## 4. 核心产品原则

### 4.1 实例是主对象
产品主视角统一围绕“实例”展开。

用户可见的实例分为两类：

* `managed`
  * 由产品创建并托管运维
* `adopted`
  * 已存在的实例，被纳入访问和诊断管理

结论：
* `Instance` 是唯一顶层产品对象
* 用户动作必须表现为“对实例做什么”

### 4.2 认证单独管理
认证配置是平台级共享资源，不应散落在各个实例表单中长期独立维护。

结论：
* `Credential` 是用户可管理的共享资源
* 但不是与 `Instance` 并列的主导航对象

### 4.3 访问与运维分离协作
ClawStation 管理访问平面，ClawOps 管理运维平面，两者协作但不混为一个执行状态机。

更具体的责任边界：

* `ClawStation`
  * 面向最终用户提供功能露出
  * 负责交互、状态、反馈和用户心智
* `ClawOps`
  * 提供后台执行和技术支撑
  * 负责把复杂运维动作收敛成可调用能力

### 4.4 失败必须可定位
所有关键动作必须提供清晰阶段反馈，并能追溯到操作日志或诊断信息。

## 5. 核心对象定义

### 5.1 Instance
实例是当前产品的主对象，至少包含以下信息：

* `id`
* `name`
* `environment`
* `sourceType`
* `managementMode`
* `deploymentTargetId`
* `host`
* `sshPort`
* `credentialId`
* `localAddress`
* `browserEntryAddress`
* `status`
* `lastError`
* 接管校验状态与摘要

### 5.2 Credential
认证配置是共享对象，至少包含：

* `id`
* `name`
* `sshUser`
* `authMethod`
* `privateKeyPath`

支持方式：

* 密码认证
* 外部私钥路径
* 托管私钥

### 5.3 Deployment Objects
以下对象属于技术支撑对象，而不是顶层产品主对象：

* `HostProfile`
* `DeploymentProfile`
* `DeploymentTarget`
* `OperationRecord`

结论：
* 这些对象不应成为最终用户心智的一部分
* 它们只应存在于后台实现、调试能力或开发辅助场景

## 6. 当前版本功能范围

### 6.1 已支持

#### 实例基础管理
* 列出实例
* 新建实例
* 编辑实例
* 删除实例
* 默认实例选择
* 按环境筛选

#### 认证管理
* 认证配置新增、编辑、删除
* SSH 密码保存/清除
* 托管私钥保存/清除
* 认证状态检查

#### 主机密钥管理
* 扫描实例主机密钥
* 扫描部署主机密钥
* 首次信任
* 密钥变更阻断

#### 访问能力
* 预览连接计划
* 建立 SSH 隧道
* 断开连接
* 打开浏览器
* 自动连接默认实例

#### 新建托管实例
* 通过实例创建流程完成主机、部署模板和托管关系准备
* 自动生成或更新托管实例
* 可选择立即部署或仅保存

#### 接管已有实例
* 创建前正式校验
* 接管失败保存为草稿
* 草稿持久化
* 已接管实例重新校验

#### 托管运维
* `deploy`
* `probe`
* `restart`
  * 仅对全新部署实例开放
* `rollback`
  * 仅对全新部署实例开放

#### 诊断与日志
* 实例诊断
* 接管校验诊断
* 实例级最近操作
* 实例级聚合日志

#### 配置管理
* 导出配置包
* 导入配置包
* 列出归档
* 恢复归档

### 6.2 当前版本不作为主能力支持
* Docker 安装和主机初始化
* 交互式 `sudo`
* Kubernetes / Docker Compose 等复杂编排
* 团队协作权限和审批流
* 公网暴露与 HTTPS 证书管理
* 浏览器内嵌 OpenClaw 页面
* 面向最终用户的 `Deployments` 视图

## 7. 当前版本能力矩阵

| 类型 | 连接/打开 | 诊断 | 日志 | 接管重校验 | 部署/探测 | 重启/回滚 |
|---|---|---|---|---|---|---|
| `managed + deployment_managed + fresh_deployed` | 支持 | 支持 | 支持 | 不适用 | 支持 | 支持 |
| `managed + deployment_managed + managed_existing` | 支持 | 支持 | 支持 | 不适用 | 支持 | 不支持 |
| `adopted + access_only` | 仅校验通过后支持 | 支持 | 支持 | 支持 | 不适用 | 当前不作为正式能力暴露 |

补充原则：
* 运维动作虽然由 `ClawOps` 执行，但用户入口必须落在实例详情中
* 用户不需要理解 `HostProfile / DeploymentProfile / DeploymentTarget`

## 8. 核心用户路径

### 8.1 新建并部署实例
1. 用户从“新建实例”进入。
2. 选择部署新实例。
3. 选择认证配置。
4. 选择或创建主机配置。
5. 选择或创建部署配置。
6. 创建托管实例。
7. 立即执行部署，或先保存待后续部署。

### 8.2 接管已有实例
1. 用户从“新建实例”进入。
2. 选择接管已有实例。
3. 填写实例信息并选择认证配置。
4. 执行 SSH、主机密钥和 HTTP 探测校验。
5. 校验成功则纳入实例列表。
6. 校验失败则允许保存草稿并后续重试。

### 8.3 连接并打开
1. 用户从实例列表选择实例。
2. 点击连接或打开。
3. 应用解析认证、校验主机密钥、建立 SSH 隧道。
4. 本地浏览器入口可用后打开默认浏览器。

### 8.4 托管实例运维
1. 用户从实例详情发起部署、探测，以及仅对全新部署实例开放的重启或回滚。
2. 应用将动作委托给 ClawOps。
3. ClawOps 记录操作、写日志并返回事件。
4. 用户可在实例详情查看操作结果和日志。

说明：
* 用户看到的是“实例运维动作”
* 不应看到“对 DeploymentTarget 执行动作”的产品表述

## 9. 当前版本状态模型

### 9.1 当前实现中的主要实例状态
当前代码主状态为：

* `idle`
* `preparing`
* `connecting`
* `forwarding`
* `probing`
* `ready`
* `degraded`
* `error`

### 9.2 接管校验状态
接管实例额外维护：

* `pending`
* `validated`
* `failed`

其作用是约束 adopted 实例在未通过校验前不能被直接访问。

## 10. 当前版本的产品边界结论

### 10.1 已经确认的稳定定义
* 实例是当前产品主对象
* 认证配置是共享资源
* 接管必须经过正式校验
* SSH 主机密钥必须严格校验
* 托管运维通过独立 ClawOps 执行
* 日志和诊断必须能追溯到实例
* `Deployments` 不面向最终用户体现

### 10.2 仍在收敛中的部分
* 实例与认证之间仍有缓存字段共存
* 目标对象 `Target` 在运维实现中仍然较强
* adopted 实例的后续运维边界还可以进一步明确

## 11. 后续文档维护规则
* 产品目标、范围和用户路径变化时，优先更新本文档
* 技术实现路线变化时，更新 `设计文档.md`
* 当前代码现状发生较大变化时，同时更新两份文档
